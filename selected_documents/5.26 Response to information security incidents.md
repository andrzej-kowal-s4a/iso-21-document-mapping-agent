# Selected Documents for ISO 27001 Control 5.26 Response to information security incidents

## 1. Security Incident Reporting Procedure / Procedura zgłaszania incydentów związanych z bezpieczeństwem
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170104051

This is the primary document for incident response procedures and directly addresses the control requirement for documented incident response procedures. It establishes the framework for reporting security incidents, defines what constitutes a security incident, and outlines the initial response steps including notification to the Security Team and creation of Jira tickets. The procedure is part of the overall Security Incident Management Plan and ensures all personnel know how to report incidents, which is fundamental to initiating the response process.

## 2. Communication procedure in case of Major Incident
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170102638

This procedure directly addresses the control requirement for communicating incidents to relevant internal and external interested parties. It establishes a clear process flow for major incidents including notification procedures, status updates, customer communication, and coordination with support teams. The procedure ensures proper communication channels are used and that all relevant parties are informed according to the need-to-know principle, which aligns with requirement (e) of the control.

## 3. Major Incidents
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/SUP/pages/160433356

This document is highly relevant as it covers escalation procedures, coordination activities, and post-incident analysis including Root Cause Analysis (RCA). It establishes procedures for managing major incidents including escalation to management, coordination with responsible parties through Slack channels, and ensuring RCA is performed. The document addresses requirement (c) for escalation and crisis management activities, requirement (f) for coordination with internal parties, and requirement (i) for post-incident analysis to identify root cause.

## 4. Personal Data Protection Breach Procedure
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170099350

This procedure is relevant because it addresses incident response activities specific to personal data breaches, including containment, assessment, notification, and coordination with customers and supervisory authorities. It establishes procedures for managing breaches that are treated as security incidents, includes risk assessment methodology, and covers coordination with external parties (customers, supervisory authorities) which aligns with requirement (f) for coordinating with external parties. The procedure also ensures proper documentation and logging of breach response activities.

## 5. Bring Your Own Device (BYOD) Policy
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170106633

This policy is relevant because it contains specific requirements for forensic evidence collection in the context of security incidents involving BYOD devices. It requires team members to cooperate and provide assistance during security incidents, including providing forensic data (logs, evidence), performing disk wiping, and installing forensic software. This directly addresses requirement (b) for collecting evidence as soon as possible after occurrence and requirement (h) for conducting forensic analysis as required.

## 6. Information Security Policy
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170104722

This policy establishes the overall framework for security incident management and references the Security Incident Management Plan. It states that Smart4Aviation Group has a Security Incident Management Plan established to effectively respond to security incidents, applicable to both internal and customer-related assets. The policy provides the foundational requirement for incident response procedures and ensures that any observed or suspected security incidents are reported, which supports the control's requirement for documented procedures and proper incident response.
