# Selected Documents for ISO 27001 Control 5.3 Segregation of duties

## 1. Information Security Policy
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170104722

This document serves as the foundation for all security controls including segregation of duties. It establishes the overarching security framework that guides how conflicting duties should be separated to prevent fraud, errors, and security bypasses as required by the control.

## 2. Change Management Policy
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170104688

This policy directly addresses one of the key examples in the control guidance: separating duties for initiating, approving, and executing changes. It establishes formal processes to ensure changes follow proper authorization workflows with appropriate segregation of duties at each step.

## 3. Procedure: Change to Baseline Management
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170106091

This procedure implements segregation of duties specifically for baseline changes, ensuring that different individuals are responsible for different aspects of the change process. It provides practical guidance for implementing segregated responsibilities in the software development lifecycle.

## 4. Server management procedure
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170107313

This document addresses the control requirement for segregating roles between developing software and administering production systems. It details how server access and management responsibilities are segregated to prevent conflicts of interest in system administration tasks.

## 5. Product Development and Support Security Credentials Procedure
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170106199

This procedure establishes how credentials are managed between development and support teams, implementing segregation of duties between designing, implementing and reviewing code, as well as separating development from production system administration as specified in the control guidance.
