# Selected Documents for ISO 27001 Control 5.3 Segregation of duties

## 1. Change Management Policy
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170104688

This policy establishes the framework for managing changes with appropriate levels of approval and authorization, ensuring that no single individual can both initiate and approve changes. It directly supports the control's requirement to segregate duties related to change management processes, addressing example (a) from the control guidance.

## 2. Information Security Policy
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170104722

The Information Security Policy establishes the overarching principles for information security, including requirements for segregation of duties across different security domains. This document provides the foundation for implementing segregated responsibilities and prevents conflicts that could lead to security control bypassing.

## 3. Server management procedure
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170107313

This procedure establishes separated roles and responsibilities for server management, ensuring that development and production system administration duties are segregated. It addresses control guidance examples (d) and (e) by preventing the same person from developing software and administering production systems.

## 4. Product Development and Support Security Credentials Procedure
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170106199

This procedure defines security credential management for development and support teams, implementing segregation between those who develop code and those who manage access to production environments. It supports control examples (c) and (d), ensuring separation between design, implementation, and review roles.

## 5. Procedure: Change to Baseline Management
**Confluence URL:** https://smart4aviation.atlassian.net/wiki/spaces/S4A/pages/170106091

This procedure establishes separation between proposing changes and approving them through the Change Review Board (CRB) process. It directly addresses example (a) in the control by ensuring that individuals who initiate changes cannot also approve and execute them without oversight.
